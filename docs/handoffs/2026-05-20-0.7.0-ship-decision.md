## Accepted Warnings And Limitations

### `repo-exposure-latency`

Disposition: accepted warning for 0.7.0; defer latency work after release.

`cargo xtask release-readiness --version 0.7.0` passed all required checks and
reported overall `warn` only because the non-required `repo-exposure-latency`
report timed out at the advisory 30s threshold. The JSON path timed out after
processing 500 of 38927 seams, and the Markdown path was skipped after the JSON
timeout. This does not block the release under the current release-readiness
contract.

### Current full `ripr-swarm` live audit

Disposition: accepted 0.7.0 limitation; do not claim live current-repo repair
proof.

The 0.7 dogfood pass recorded that the current full `ripr-swarm` audit timed
out before producing current-repo actionable packets. That is acceptable for
0.7.0 because the release claim is the actionable static repair loop over
existing typed artifacts. The timeout fails closed and does not create a fake
repair packet.

### VS Code dependency audit

Disposition: accepted warning for 0.7.0 package proof; open a separate
dependency lane if maintainers want to remediate before a later marketplace
update.

`npm --prefix editors/vscode ci` passed but reported three audit findings in
the existing VS Code dependency tree. This packet does not run `npm audit fix`,
change dependencies, or modify lockfiles.

### VSIX bundle-size warning

Disposition: accepted packaging warning for 0.7.0; defer bundling/file-count
cleanup.

`npm --prefix editors/vscode run package` passed and produced
`editors/vscode/dist/ripr-0.7.0.vsix`. `vsce` reported the existing JavaScript
file-count bundling warning.

## Release Claims Allowed

0.7.0 public copy may claim:

- `ripr` turns existing static test-gap evidence into bounded actionable repair
  packets;
- `cargo xtask ripr-swarm plan --top <n>` ranks existing actionable packets
  into swarm-ready and blocked groups;
- `cargo xtask ripr-swarm attempt --packet <id> --dry-run` prints one bounded
  repair context and remains source-edit-free;
- the repair packet names canonical gap identity, repair route, related test or
  observer, verify command, receipt command, confidence basis, supporting raw
  findings, and must-not-change boundaries when those typed fields exist;
- outcome reporting distinguishes not attempted, improved, unchanged,
  regressed, resolved, attempted-without-receipt, and orphaned receipt states;
- editor, first-pr, and PR/CI surfaces can project typed actionable repair
  packets without becoming analyzers, agents, or policy engines.

0.7.0 public copy must not claim:

- autonomous repair;
- source-edit automation;
- generated tests;
- provider or model calls;
- runtime mutation execution;
- killed or survived mutation outcomes;
- runtime adequacy;
- coverage adequacy;
- correctness proof;
- default CI blocking;
- gate eligibility by default;
- TypeScript, JavaScript, or Python preview promotion;
- current full `ripr-swarm` live audit completion or live current-repo repair
  selection.

## Exact Next Execution Steps

These are the next release-operator steps. They are intentionally split across
explicit PRs so publish, server assets, marketplace surfaces, badge endpoints,
and release closeout remain separately reviewable.

### 1. Prepare the local release checkout

```bash
git fetch origin main --tags
git checkout main
git pull --ff-only origin main
git rev-parse HEAD
git status --short
```

Expected `HEAD` should be the latest `origin/main` commit that contains this
ship-decision handoff and no later code, release-surface, policy,
generated-evidence, dependency, or badge change unless a maintainer explicitly
accepts the newer release base and reruns the required proof.

### 2. Confirm final no-drift gates

```bash
cargo xtask check-generated-clean
cargo xtask release-readiness --version 0.7.0
cargo publish -p ripr --dry-run
npm --prefix editors/vscode run package
git diff --check
```

Abort if any required release-readiness check fails, if generated residue is
tracked, or if the package/VSIX dry run no longer passes.

### 3. Publish crates.io in the next execution PR

```bash
cargo login
cargo publish -p ripr
```

If Cargo times out while polling after upload, check crates.io manually before
retrying. Do not run a second publish until registry state is known.

The publish PR must record:

- source commit;
- final dry-run proof;
- crates.io publish result;
- package version proof;
- no source diff after publish.

### 4. Package server and VSIX artifacts in the following PR

After the crate publish result is recorded, package release server and VSIX
artifacts explicitly:

```bash
npm --prefix editors/vscode run package
```

If GitHub Release assets are produced by the normal release workflow, trigger
and verify that workflow in the server/VSIX artifact PR, not in this decision
packet.

### 5. Publish editor packages only if intended

If 0.7.0 should update marketplace surfaces, publish VS Code Marketplace and
Open VSX artifacts in a separate explicit PR or release execution step. If not,
record that the crate release shipped first and marketplace publication is
deferred.

### 6. Close release execution

After publish/artifact work completes, add a release execution closeout that
records published crate version, editor package status, GitHub Release status,
artifact checksums, release notes, known limitations, accepted non-claims,
source commit, tag status, validation commands, and follow-up issues.
